Chapter 1

What Ginie actually is.

Before understanding how it works, understand what it is. Ginie is a language model — a giant mathematical function that takes text in and produces text out. That is it. Everything else is just implementation detail.

Concept 01
Language Model
A language model predicts the most likely next word given everything before it. That is the only thing it does — over and over — until it builds a complete response. Ginie predicts the next token in a Solidity contract based on the instruction you gave it and all the tokens it has already generated.
Analogy: Predictive text on your phone, but trained on millions of smart contracts instead of your messages.
Concept 02
Parameters
Ginie has 2 billion parameters. A parameter is just a number. The model is literally 2 billion numbers arranged in matrices. During training, those numbers get adjusted until the model produces good contracts. After training, they are frozen — the numbers never change again during inference (when you use the model).
Analogy: A very complicated dial with 2 billion knobs. Training is turning all the knobs to the right position. Using the model is just reading the output with the knobs fixed.
Concept 03
Fine-tuning vs training from scratch
Ginie did not start from zero. It started from Chain-GPT/Solidity-LLM, which started from Salesforce/codegen-2B-multi. That base model was trained on billions of lines of code from the entire internet. Fine-tuning just adjusts the existing knowledge toward smart contracts specifically — it takes hours instead of months.
Analogy: Hiring a senior developer and giving them a 2-week Solidity bootcamp. You do not retrain their entire brain.
Concept 04
Inference
Inference means using the model — giving it a prompt and getting output. Training is when the model learns. Inference is when you actually use what it learned. The HuggingFace Space runs inference. Colab training runs training. They are completely different processes requiring completely different amounts of compute.
Analogy: Training is studying for an exam. Inference is taking the exam. Studying takes weeks. Taking the exam takes an hour.
Chapter 2

The tokenizer.
How text becomes numbers.

The model cannot read text. It only works with numbers. The tokenizer converts your text into numbers before the model sees it, and converts numbers back to text after the model finishes. Ginie uses the GPT-2 tokenizer.

1
You write a prompt
You type: "Write a Solidity ERC20 token with minting and burning." This is raw text — the model cannot process this directly.
"Write a Solidity ERC20 token with minting and burning."
2
Tokenizer splits text into tokens
The tokenizer does not split by word. It splits by frequency — common words become one token, rare words get split into pieces. "Solidity" might be one token. "minting" might split into "mint" + "ing".
["Write", " a", " Sol", "idity", " ERC", "20", " token", " with", " minting", " and", " burning", "."]
3
Each token gets an ID number
Every token in the vocabulary has a unique integer ID. "Write" might be 16289. " a" might be 257. The tokenizer looks up each token and replaces it with its number.
input_ids = [16289, 257, 1885, 1799, 23485, 1238, 11241, 351, 20350, 290, 6533, 13]
4
Model processes the numbers
The model works entirely with these integer IDs — it never sees text. It produces new integer IDs as output, one at a time, until it generates the EOS (end of sequence) token or hits max_new_tokens.
output_ids = [198, 1303, 50, 47, 28, 55, 25, 198, 28, 36, 2926, ...] # the contract
5
Tokenizer decodes numbers back to text
tokenizer.decode() looks up each output ID in the vocabulary and converts it back to the corresponding text. You get readable Solidity code.
tokenizer.decode(outputs[0], skip_special_tokens=True) # → "// SPDX-License-Identifier: MIT..."
What is max_new_tokens?
This is the maximum number of new tokens the model can generate in one response. In your app you set it to 600–800. One token is roughly 0.75 words on average. So 800 tokens ≈ 600 words ≈ about 40–60 lines of Solidity. If a contract needs more than that, the model cuts off mid-way. For complex contracts, increase to 1200–1500.
Vocabulary size
The GPT-2 tokenizer has a vocabulary of 50,257 tokens. That means 50,257 unique "words" (actually subword pieces). Every piece of text you could possibly write can be represented as a sequence of those 50,257 pieces. The model never sees anything outside that vocabulary.
Chapter 3

The transformer.
How the model thinks.

The transformer is the architecture — the structure — of the model. Ginie has 32 transformer layers stacked on top of each other. Each layer refines the model's understanding of the input a little more. By layer 32, it knows what to write next.

Layer structure
What happens in each of the 32 layers
Each transformer layer does two things: first, self-attention (every token looks at every other token and decides which ones matter), then a feed-forward network (each token individually processes its updated representation). After 32 of these, the model has a rich understanding of context and produces the next token.
Why 32 layers?
Depth = better reasoning
Each layer can learn something the previous layer missed. Layer 1 might learn basic syntax. Layer 5 might learn variable relationships. Layer 20 might learn security patterns. Layer 32 synthesises all of it to predict the next token. More layers = more sophisticated understanding — but also more compute needed.
Embeddings
Token IDs become vectors
When token ID 16289 enters the model, the embedding layer converts it into a vector — a list of 2048 numbers. These numbers capture the semantic meaning of the token. The word "transfer" in a Solidity context gets a different vector than "transfer" in an English essay. This contextual meaning is what the transformer layers refine.
Key number: Ginie's embedding dimension is 2048. Each token is represented as a list of 2048 floating-point numbers throughout the model.
Context window
2048 tokens max at once
Ginie can hold 2048 tokens in its "working memory" at a time. That includes your prompt AND everything it has generated so far. If the prompt + output exceeds 2048 tokens, the model starts forgetting the beginning. This is the context length limitation — a key constraint for long contracts.
Chapter 4

K, Q, V explained.
The heart of attention.

Inside every transformer layer, there is a self-attention mechanism. It uses three matrices called Query, Key, and Value. This is what allows the model to understand relationships between tokens — how "transfer" relates to "amount" and "address" in a Solidity function, for example.

Q — Query
What am I looking for?
Each token generates a Query vector — essentially a question it is asking about the context. The token "balanceOf" is asking: "which other tokens in this sequence are relevant to understanding what I mean?"
Analogy: A search query. "Show me things related to token balance."
K — Key
What do I have to offer?
Each token also generates a Key vector — an advertisement of what it contains. The token "address" advertises itself as relevant to wallet locations, function parameters, and ERC20 operations.
Analogy: The metadata on a file. "I contain: address-related information."
V — Value
My actual content
The Value vector is the actual information that gets passed along if the attention score is high. Once Q and K determine that two tokens should attend to each other, the V of the relevant token gets weighted into the output.
Analogy: The contents of the file. Once you find the right file (via Q·K match), you read its contents (V).

"transfer" attends strongly to "address" and "amount" — it knows these are its arguments. Bar width = attention weight.
Multi-head attention
Ginie does not run attention once per layer — it runs it 16 times in parallel, each with different Q/K/V matrices. Each "head" learns to attend to different relationship types. One head might specialise in syntax. Another in security patterns. Another in variable scoping. The outputs of all 16 heads are concatenated and mixed. 16 attention heads × 32 layers = 512 attention operations per forward pass.

Temperature — what it does
At each step, the model produces a probability distribution over all 50,257 tokens. Temperature scales those probabilities. Temperature 0: always pick the most likely token (deterministic, repetitive). Temperature 1.0: pick randomly from the distribution (creative, sometimes wrong). Temperature 0.7: the sweet spot — mostly follows the most likely path but introduces controlled variation. Ginie uses 0.7 by default.
do_sample vs greedy
do_sample=True means the model samples from the probability distribution — it does not always pick the most likely next token. This gives varied, more natural outputs. do_sample=False (greedy decoding) always picks the single most likely token — outputs are deterministic but often loop or become repetitive. Always use do_sample=True with temperature 0.6–0.8 for code generation.
Chapter 6

Training on Colab.
Step by step.

Google Colab gives you a free NVIDIA T4 GPU with 16GB VRAM. That is enough to fine-tune a 7B model using LoRA (Low-Rank Adaptation) — the technique that makes training fast and cheap. Here is exactly how it works.

1
Install Unsloth — the training framework
Unsloth is a library that makes LoRA fine-tuning 2–5× faster and uses 60% less memory than vanilla HuggingFace training. It is specifically optimised for T4 and Apple Silicon.
!pip install unsloth transformers datasets trl torch accelerate
2
Load the base model in 4-bit precision
4-bit quantisation shrinks the model from 14GB to about 4GB in memory. The model loses almost no quality but now fits on a free T4. load_in_4bit=True is the key setting.
from unsloth import FastLanguageModel model, tokenizer = FastLanguageModel.from_pretrained( "GinieAI/Solidity-LLM", max_seq_length=2048, load_in_4bit=True # 14GB → 4GB )
3
Add LoRA adapters
LoRA adds tiny trainable matrices (A and B) alongside the frozen model weights. Only A and B update during training — the 2B original parameters stay frozen. rank=16 means the matrices have 16 columns — a small but effective update size. This is why training takes 25 minutes instead of 1,095 hours.
model = FastLanguageModel.get_peft_model( model, r=16, # LoRA rank — higher = more capacity lora_alpha=16, # scaling factor target_modules=["q_proj","k_proj","v_proj","o_proj"], )
4
Format your training data
Every training example must follow the exact same prompt format as your inference. The model learns to complete the Response section given an Instruction. Consistency here is critical — if your training format differs from your inference prompt, the model outputs garbage.
def format(example): return {"text": f"""### Instruction: {example['instruction']} ### Response: {example['output']}"""} dataset = dataset.map(format)
5
Train with SFTTrainer
SFTTrainer (Supervised Fine-Tuning Trainer) is the standard training loop from HuggingFace TRL. You set the number of epochs (3 is usually right for this size dataset), batch size (4 works on T4), and learning rate (2e-4 is safe for LoRA). save_steps=25 means a checkpoint saves every 25 steps — if Colab disconnects, you do not lose everything.
from trl import SFTTrainer trainer = SFTTrainer( model=model, tokenizer=tokenizer, train_dataset=dataset, dataset_text_field="text", max_seq_length=2048, args=TrainingArguments( num_train_epochs=3, per_device_train_batch_size=4, learning_rate=2e-4, save_steps=25, # save checkpoint every 25 steps output_dir="checkpoints", ) ) trainer.train()
6
Save and push to HuggingFace
Save the adapter (not the full model — the adapter is just the tiny LoRA matrices, a few hundred MB). Push to HuggingFace so it is safe even if Colab shuts down. The merged model (adapter + base weights) is what you deploy.
model.save_pretrained("ginie-v2-adapter") model.push_to_hub("GinieAI/Solidity-LLM-v2") # push adapter to HF # To merge and push the full model: merged = model.merge_and_unload() merged.push_to_hub("GinieAI/Solidity-LLM-v2-merged")
Critical Colab warning
Colab free sessions disconnect after 3–12 hours of inactivity. Always save to Google Drive or push to HuggingFace Hub every 25 steps. If the session dies mid-training, you resume from the last checkpoint. Set save_steps=25 in your TrainingArguments — this is non-negotiable for free tier Colab.
What the loss number means
During training you will see a number called "loss" printed every few steps. It starts high (around 2.5) and should decrease toward 0.6–1.0 after 3 epochs. If loss does not decrease, the data format is wrong — check that your prompt format exactly matches the format the base model was trained with. Fix data quality before changing any hyperparameter.

Chapter 8

What it can do.
What it cannot.

Honest assessment of the current v1 model. Knowing the limitations is as important as knowing the strengths — because the limitations tell you exactly where to focus your v2 training data.

What Ginie v1 does well
  • Standard ERC20 contracts — mint, burn, transfer, allowance
  • ERC721 NFT contracts with basic minting
  • OpenZeppelin import patterns — uses the right library for each standard
  • Solidity >= 0.8 syntax — modern Solidity with SafeMath built in
  • Basic access control — Ownable, Role-based with OZ
  • Events, modifiers, constructor patterns
  • Simple staking and reward contracts
  • Following the instruction format consistently
Where Ginie v1 struggles
  • Complex DeFi — AMMs, liquidity pools, yield farming with multi-token logic
  • Daml contracts — not yet in v1, planned for v3
  • Reentrancy guards — the 58% security score reflects this gap
  • Long contracts over ~150 lines — hits context window limits
  • Cross-contract interactions — calling external contract interfaces reliably
  • Proxy and upgradeable contract patterns
  • Gas optimisation at the assembly level
  • Rare or niche standards like ERC4626, ERC2981
The 58% security score — what it means in practice
58% security means Slither finds issues in about 42% of generated contracts. The most common issues are: missing reentrancy guards on external calls, unchecked return values from external calls, and integer overflow in edge cases (even with Solidity 0.8's built-in checks, some patterns still need explicit handling). Never deploy a Ginie-generated contract to mainnet without running Slither and fixing every high-severity finding.
Chapter 9

How the HuggingFace
Space works.

The Space runs Gradio on a free CPU tier inside a Docker container on HuggingFace's servers. Understanding the architecture explains why it is slow and how to make it faster.

1
Docker container starts on HuggingFace servers
When someone opens the Space, HuggingFace spins up a Docker container running Ubuntu. It installs the packages from requirements.txt — gradio, transformers, torch, accelerate. This happens once per deployment and takes 2–4 minutes.
2
app.py runs — model downloads from HuggingFace Hub
The first time app.py runs, it calls AutoModelForCausalLM.from_pretrained("GinieAI/Solidity-LLM"). This downloads 6GB of model weights from the HuggingFace CDN into the container's local cache. On a cold start (no cache) this takes 3–5 minutes. On a warm start (cached), 30 seconds.
model = AutoModelForCausalLM.from_pretrained( "GinieAI/Solidity-LLM", torch_dtype=torch.float32, # float32 because no GPU — CPU only device_map="cpu" )
3
Gradio serves the UI
Gradio is a Python library that creates the web interface — the text boxes, sliders, and buttons — with one decorator. demo.launch() starts a web server on port 7860. HuggingFace proxies external traffic to that port. The user's browser loads the Gradio frontend (HTML/JS) and communicates via WebSocket.
4
User submits — generate_contract() runs on CPU
The generate_contract() function runs entirely on CPU — no GPU available on the free tier. CPU inference is 20–50× slower than GPU. A 600-token output that would take 8 seconds on a T4 takes 3–8 minutes on CPU. This is the main performance problem with the current Space.
with torch.no_grad(): outputs = model.generate( input_ids=inputs["input_ids"], attention_mask=inputs["attention_mask"], max_new_tokens=600, # each of these takes ~0.3s on CPU temperature=0.7, do_sample=True, pad_token_id=tokenizer.eos_token_id, )
5
Response streams back to the browser
The generated text is decoded and returned to Gradio, which displays it in the Code component. The current implementation returns everything at once (not streaming). Adding a TextStreamer would show tokens appearing in real time — much better user experience.

Chapter 10

How to make it better.
The exact steps.

Every improvement mapped to what it fixes, how to do it, and what result to expect. In order of impact.

01
Fix the Space build error — gradio version conflict
Change requirements.txt to pin gradio==4.44.0 and transformers==4.41.2. The current 6.10.0 version has dependency conflicts on HuggingFace's free tier Docker environment. This is why the Space is broken right now.
gradio==4.44.0 transformers==4.41.2 torch==2.2.2 accelerate==0.30.0
02
Add streaming to the Space — users see output appearing in real time
Replace the current generate call with TextIteratorStreamer. Instead of waiting 5 minutes to see anything, users watch the contract appear token by token. This alone makes the Space feel 10× faster even if the total generation time is identical.
from transformers import TextIteratorStreamer from threading import Thread streamer = TextIteratorStreamer(tokenizer, skip_prompt=True, skip_special_tokens=True) thread = Thread(target=model.generate, kwargs={**inputs, "streamer": streamer, ...}) thread.start() for chunk in streamer: yield chunk # Gradio streams this to the UI
03
Retrain v2 on 7,800 examples to push security score above 70%
The 58% security score is the main weakness in the benchmark. Fix it by adding the mwritescode/slither-audited dataset (zero high-severity contracts only) and darkknight25/vulnerability dataset as negative examples. The model learns what good security looks like AND what bad security looks like.
# Filter slither-audited for zero high-severity findings safe_contracts = [c for c in slither_data if c["high_severity_count"] == 0] # Add vulnerability examples as "bad output" negative examples # Merge into 7,800 total training examples
04
Add daml-finance training data — unlock the Canton moat
Clone digital-asset/daml-finance. For every .daml template file, use Claude API to generate an instruction-output pair. This produces ~200 high-quality Daml examples that no other model on HuggingFace has been trained on. Merge into the training set. Ginie becomes the only model that can write Daml.
import subprocess, os subprocess.run(["git", "clone", "https://github.com/digital-asset/daml-finance"]) daml_files = [f for f in os.listdir("daml-finance/src") if f.endswith(".daml")] for f in daml_files: content = open(f).read() instruction = claude_api(f"Generate instruction for this Daml contract:\n{content}") # Save as JSONL training pair
05
Export GGUF — multiply downloads by 10×
The GGUF format runs on any laptop without a GPU via llama.cpp or Ollama. The audience for "runs locally on my Mac" is 10× larger than the audience for "runs on CUDA only." Uploading a Q4_K_M quantised GGUF alongside the safetensors immediately reaches every developer running an M-series Mac.
python llama.cpp/convert_hf_to_gguf.py GinieAI/Solidity-LLM --outtype f16 ./llama.cpp/llama-quantize ginie-f16.gguf ginie-q4_k_m.gguf Q4_K_M # Upload to GinieAI/Solidity-LLM-GGUF on HuggingFace
06
Build the data flywheel — the permanent moat
Log every prompt, output, compile status, and security score in production. Every Friday, filter for: compiled=True AND security_score >= 70. Add these to the training set. Retrain v(n+1). The model improves weekly from real user prompts — a data distribution competitors cannot buy or synthesise because they do not have your users.
import json, datetime def log_generation(prompt, output, compiled, security_score): entry = { "instruction": prompt, "output": output, "compiled": compiled, "security_score": security_score, "timestamp": datetime.datetime.utcnow().isoformat() } with open("logs.jsonl", "a") as f: f.write(json.dumps(entry) + "\n")
